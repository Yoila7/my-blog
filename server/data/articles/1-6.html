Web渲染过程

DNS查询

DNS（Domain Name System）

域名系统。仅知道域名并不能直接对应上服务器，需要通过DNS将域名解析为IP地址。如果多次访问同一域名，会优先查询本地缓存中是否已经保存了域名对应的IP地址。

IP地址

一串用“.”分隔的数字，代表着接入网络的一台具体的计算机。访问网站意味着需要从网站对应的服务器获取信息，而找到这一具体的服务器就需要知道它的IP地址。

域名

如“google.com”。域名是为IP地址取的名字，一般通过域名访问网站，因为服务器的IP地址可能是动态的，而且同一网站可能有多个服务器。

二者联系

  • 一对一：一个域名指向一个IP地址
  • 一对多：大型网站访问量大，因此域名会指向多个服务器，通过域名访问一般会连接到最接近或最空闲的服务器。
  • 多对一：小型网站服务器负载不大，可以多个网站共用一个服务器。

TCP握手

TCP（Transmission Control Protocol）传输控制协议

获取服务器IP地址后，浏览器通过TCP三次握手与服务器连接，用于协商连接参数。 三次握手分别为SYN、SYN-ACK、ACK。

synchronize同步/acknowledgement确认

TLS协商

TLS（Transport Layer Security）传输层安全协议

通过HTTPS进行安全连接，还需要经过TLS协商，确定加密方式，验证服务器，共需要五次传输。

拥塞控制/TCP慢启动

传输过程中，数据包被分割成段，传输一定数量的段后，客户端需要返回一个ACK包来确认传输情况。每次确认对应的传输段数过多或过少都会产生问题，因此需要用到TCP慢启动算法。先定一个初值，若传输正常，则翻倍，否则减半（类似倍增算法？）。

解析

“解析”是浏览器将通过网络接收的数据转换为 DOM 和 CSSOM 的步骤，通过渲染器在屏幕上将它们绘制成页面。

构建DOM树

将HTML代码中的每一个元素作为节点，嵌套关系作为父子关系，可以构建一个DOM树，节点越多，构建时间越长。

< script > 标签会阻塞渲染进程，因此过多的脚本会让页面加载变慢。

预加载扫描器

浏览器构建DOM树的同时，预扫描加速器会解析可用内容并请求高优先级的资源。这种提前请求的方式会减少后续可能产生的阻塞。

构建CSSOM树

浏览器遍历 CSS 中的每个规则集，根据 CSS 选择器创建具有父、子和兄弟关系的节点树。浏览器从适用于节点的最通用规则开始，并通过应用更具体的规则递归地优化计算的样式。

构建 CSSOM 非常快，在 web 性能优化方面，它是可轻易实现的。

JavaScript编译

JavaScript 会被解析、编译和解释。脚本被解析为抽象语法树。有些浏览器引擎会将抽象语法树输入编译器，输出字节码。这就是所谓的 JavaScript 编译。

渲染

渲染步骤包括样式、布局、绘制，在某些情况下还包括合成。在解析步骤中创建的 CSSOM 树和 DOM 树组合成一个渲染树，然后用于计算每个可见元素的布局，然后将其绘制到屏幕上。

布局

在渲染树上运行布局以计算每个节点的几何体。布局是确定呈现树中所有节点的尺寸和位置，以及确定页面上每个对象的大小和位置的过程。重排是后续过程中对页面的任意部分或整个文档的大小和位置的重新计算。

绘制

在绘制或光栅化阶段，浏览器将在布局阶段计算的每个盒子转换为屏幕上的实际像素。绘制涉及将元素的每个可见部分绘制到屏幕上，包括文本、颜色、边框、阴影以及按钮和图像等替换元素。浏览器需要以超快的速度执行这个过程。

绘制可以将布局树中的元素分解为多个层。将内容提升到 GPU 上的层（而不是 CPU 上的主线程）可以提高绘制和重新绘制性能。

分层确实可以提高性能，但在内存管理方面成本较高，因此不应作为 Web 性能优化策略的过度使用。

交互

主线程绘制页面完成，并不代表已经可以开始交互。如果加载包括正确延迟加载的 JavaScript，并且仅在 onload 事件触发后执行，那么主线程可能会忙于执行脚本，无法用于滚动、触摸和其他交互操作。

Web性能优化

性能优化地图

度量标准与设定目标

度量标准

  • 首次有效绘制（First Meaningful Paint，简称FMP，当主要内容呈现在页面上）
  • 英雄渲染时间（Hero Rendering Times，度量用户体验的新指标，当用户最关心的内容渲染完成）
  • 可交互时间（Time to Interactive，简称TTI，指页面布局已经稳定，关键的页面字体是可见的，并且主进程可用于处理用户输入，基本上用户可以点击UI并与其交互）
  • 输入响应（Input responsiveness，界面响应用户输入所需的时间）
  • 感知速度指数（Perceptual Speed Index，简称PSI，测量页面在加载过程中视觉上的变化速度，分数越低越好）
  • 自定义指标，由业务需求和用户体验来决定。

设定目标

  • 100毫秒的界面响应时间与60FPS
  • 速度指标（Speed Index）小于1250ms
  • 3G网络环境下可交互时间小于5s
  • 重要文件的大小预算小于170kb

计算机网络基础知识

TCP/IP四层模型

TCP/IP四层模型是目前被广泛采用的一种网络分层模型，由以下4层组成

  • 1. 应用层(Application Layer)：这是网络通信的顶层，负责处理应用程序和用户之间的通信。
  • 2. 传输层(Transport Layer)：传输层负责管理数据的传输，确保数据从源到目的地的可靠传输。
  • 3. 网络层(Network Layer)：网络层处理数据包的路由和转发，确保数据包从一个网络节点到达另一个网络节点。
  • 4. 数据链路层(Link Layer)：数据链路层负责将数据包分成帧，并在物理媒体上传输这些帧。它还负责错误检测和校正，以确保数据的完整性。

IP地址、子网掩码和路由

IP地址是在网络中唯一标识设备的地址，分为IPv4和IPv6。

子网掩码用于划分网络中的子网，它决定了哪些部分是网络地址，哪些部分是主机地址。

路由是决定数据包如何在不同网络之间传输的过程，通常由路由器来完成。

端口和协议

端口

你可以将端口看作是计算机上的门，每个门都通向不同的应用程序或服务。计算机有许多端口，用于不同的目的。 比如，80号端口通常是Web服务器的门，它用于接收Web浏览器发送的请求。当你在浏览器中输入网址时，浏览器会使用80号端口与Web服务器建立连接，这样你就能够看到网页内容。 所以，端口就是一种标识不同应用程序或服务的方式，它们帮助数据在计算机之间流动，确保每个应用程序都能够准确地收到它需要的数据。

常见端口和它们的用途

  • 80端口：HTTP通信。
  • 443端口：HTTPS通信。
  • 25端口：SMTP邮件传输。
  • 22端口：SSH远程登录。
  • 21端口：FTP文件传输。

协议与端口的关系

协议规定了数据包的格式和通信方式，每个协议通常使用特定的端口进行通信，以确保数据包到达正确的应用程序。

HTTP/HTTPS协议

HTTP（Hypertext Transfer Protocol）是一种用于在Web上传输数据的协议。HTTP请求和响应是Web通信的基础。

HTTP请求和响应结构

请求由以下几部分组成

  • 请求方法（Method）：常见的方法有GET（获取数据）、POST（提交数据）、PUT（更新数据）等。
  • 请求URL（Uniform Resource Locator）：指示了要访问的资源的位置。
  • 请求头（Headers）：包括各种元数据，如浏览器类型、所需语言等。
  • 请求体（Body）：用于传输一些数据，通常在POST请求中使用。

HTTP响应包括以下部分

  • 状态码（Status Code）：常见的状态码有200（成功）、404（未找到）、500（服务器错误）等。
  • 响应头（Headers）：包含了与响应相关的信息。
  • 响应体（Body）：包含实际的数据，如网页内容或API返回的数据。

HTTPS的加密机制

HTTPS（Hypertext Transfer Protocol Secure）是HTTP的加密版本，用于安全传输数据。它通过以下方式保护数据

  • 加密数据传输： HTTPS使用加密算法，将从客户端到服务器的数据加密。这意味着，即使有人截获了数据包，他们也无法轻易解密其中的内容。
  • 身份验证： HTTPS还通过数字证书来验证服务器的身份。这确保了你连接的是正确的服务器，而不是恶意的伪装者。
  • 数据完整性： HTTPS还使用消息摘要算法来确保数据在传输过程中没有被篡改。如果数据在传输过程中被修改，接收方将会意识到，并且可以拒绝处理损坏的数据。

常见HTTP状态码

状态码是服务器在处理请求时返回的三位数字代码，用于指示请求的处理结果。以下是一些常见的HTTP状态码以及它们的含义

  • 200 OK： 这表示服务器成功处理了请求，并返回了请求的内容。这是最常见的状态码，表示一切正常。
  • 404 Not Found： 这表示服务器无法找到请求的资源。通常是因为URL错误或资源被删除了。
  • 500 Internal Server Error： 这表示服务器在处理请求时发生了内部错误。这可能是服务器配置问题或应用程序错误。
  • 302 Found (重定向)： 这表示请求已被临时重定向到其他位置。浏览器会自动跟随重定向。
  • 401 Unauthorized： 这表示请求需要身份验证，但未提供有效的身份验证信息。

TCP和UDP协议

TCP和UDP是两种不同的传输层协议，适用于不同的应用场景。

TCP和UDP的特点比较

TCP是面向连接的，可靠性高，适用于需要可靠数据传输的应用。TCP通过确认和重传机制确保数据的可靠传输，同时使用滑动窗口来进行流量控制，以避免网络拥塞。

UDP是面向无连接的，传输速度快，适用于实时性要求较高的应用。UDP不具备TCP的可靠性，但适用于音视频流媒体、在线游戏等对实时性要求较高的应用。

Cookies

HTTP Cookie（也叫 Web Cookie 或浏览器 Cookie）是服务器发送到用户浏览器并保存在本地的一小块数据。浏览器会存储 cookie 并在下次向同一服务器再发起请求时携带并发送到服务器上。通常，它用于告知服务端两个请求是否来自同一浏览器——如保持用户的登录状态。

Cookie 主要用途

  • 1. 会话状态管理
  • 2. 个性化设置
  • 3. 浏览器行为跟踪
